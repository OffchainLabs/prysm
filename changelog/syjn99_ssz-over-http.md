### Added

- `--engine-ssz-http` feature flag (off by default) scaffolding the REST + SSZ Engine API v2 transport ([execution-apis#793](https://github.com/ethereum/execution-apis/pull/793)). No behavior yet; JSON-RPC `engine_*` remains the default transport.
- SSZ-over-HTTP `GET /engine/v2/identity` support (`engine_getClientVersionV1` equivalent), behind `--engine-ssz-http`.
- SSZ-over-HTTP `POST /engine/v2/{fork}/payloads` (`engine_newPayload` equivalent) for the Amsterdam/Gloas envelope, behind `--engine-ssz-http`.
- SSZ-over-HTTP `POST /engine/v2/{fork}/forkchoice` (`engine_forkchoiceUpdated` equivalent) for the Osaka/Fulu and Amsterdam/Gloas forks, behind `--engine-ssz-http`. The opaque `payload_id` is echoed verbatim; problem+json errors map onto the existing forkchoice/attribute sentinels.
- SSZ-over-HTTP `GET /engine/v2/{fork}/payloads/{id}` (`engine_getPayload` equivalent) for the Osaka/Fulu and Amsterdam/Gloas forks, behind `--engine-ssz-http`. The opaque id is hex-encoded into the path and the response is never cached.
- SSZ-over-HTTP `POST /engine/v2/blobs/v1` and `/blobs/v2` (`engine_getBlobsV1`/`engine_getBlobsV2` equivalents), behind `--engine-ssz-http`. `204 No Content` is treated as "EL cannot serve"; results are request-aligned.
- SSZ-over-HTTP `POST /engine/v2/{fork}/bodies/hash` and `GET /engine/v2/{fork}/bodies?from&count` (`engine_getPayloadBodiesByHash/ByRange` equivalents) for the Osaka/Fulu and Amsterdam/Gloas forks, behind `--engine-ssz-http`. Blinded-block reconstruction and the Gloas ePBS payload-bodies (V2) fetch now run through the engine transport; JSON-RPC behavior is unchanged.
- [Security] The SSZ-over-HTTP engine client now caps the response body read from the EL (by `Content-Length` when present and by bytes read in all cases, default 512 MiB), so a crafted or oversized response cannot coerce an unbounded allocation ([execution-apis#793](https://github.com/ethereum/execution-apis/pull/793) Security considerations).
- [Security] The SSZ-over-HTTP engine transport now respects the EL's advertised `limits.*` capabilities: `/bodies` requests are split into sub-batches of at most `bodies.max_count`, and `/blobs` (over `blobs.max_versioned_hashes`) and `/payloads` (over `payload.max_bytes`) requests are rejected before sending rather than incurring a round-trip `413`.
